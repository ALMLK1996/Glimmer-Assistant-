## Overview

Glimmer's governance structure is designed to ensure that the project remains true to its mission: **serving human beings, not extracting from them.** As the project grows, governance ensures that no single party — including the founder — can unilaterally compromise user rights or the project's ethical commitments.

## Target Governance Structure (Post-Launch)

### 1. Project Core Team

The Core Team manages day-to-day development, security, and operations.

| Role | Responsibilities |
|------|----------------|
| **Project Lead** | Strategic direction; final technical decisions; public representation |
| **Privacy Officer** | Privacy policy; data governance; user rights requests |
| **Security Officer** | Security standards; incident response; audit coordination |
| **Legal Officer** | Regulatory compliance; authority requests; legal review |
| **Community Lead** | Contributor relations; Soul marketplace; community governance |

---

### 2. Independent Oversight Board

**Purpose:** Provide external accountability for decisions affecting user rights, ethics, and compliance.

**Planned Composition:**

| Role | Qualification Required |
|------|----------------------|
| Legal Expert | International data protection law (GDPR, CCPA, or equivalent) |
| Cybersecurity Expert | Minimum 5 years professional security experience |
| AI Ethics Researcher | Academic or professional expertise in AI ethics |
| Data Protection Specialist | Certified DPO or equivalent |
| User Representative | Elected by the Glimmer user community |

**Authority of the Oversight Board:**
- Review any feature that involves new data collection
- Review all authority requests (on a confidential basis)
- Block features that violate the AI Ethics Charter
- Commission independent audits
- Publish independent annual reports

**Decisions the Oversight Board can block:**
- Any change to the Privacy Policy that weakens user protections
- Any new data collection category
- Any cooperation with authorities beyond legal requirements
- Any feature that the board determines violates the AI Ethics Charter

**Formation:** Targeted for Phase 5 (Month 7-9 post-launch)

## Decision-Making Framework

### Levels of Decision

| Level | Who Decides | Consultation Required |
|-------|------------|----------------------|
| **Routine** (bug fixes, minor features) | Core Team | None required |
| **Significant** (new features, policy changes) | Core Team | Community notification (7 days) |
| **Major** (new data collection, policy changes affecting rights) | Core Team + Oversight Board | Community consultation (30 days) |
| **Constitutional** (changes to Ethics Charter or User Rights Charter) | Core Team + Oversight Board + Community supermajority | 60-day consultation + 2/3 approval |

### Constitutional Changes

The following documents are **Constitutional** — changing them requires the highest level of process:

1. **AI_ETHICS.md** — The Seven Prohibitions cannot be weakened
2. **USER_RIGHTS.md** — User rights cannot be reduced
3. **This document** — Governance cannot be made less accountable

## Open Source Governance

As an AGPL-3.0 project, Glimmer's governance includes the broader open-source community:

### Contributor Rights
- All contributors may raise concerns through GitHub Issues
- Ethics concerns are labeled `ethics-review` and receive mandatory Core Team response
- Security concerns are handled via the vulnerability disclosure program (SECURITY.md)
- All contributors must agree to the Code of Conduct and Contributor License Agreement

### Forking Rights
The AGPL-3.0 license guarantees that anyone may fork Glimmer. However:
- Forks that remove user protections cannot use the Glimmer name or Soul format brand
- The GlimmerSoul format (CC0) may be used by anyone, including forks
- We encourage forks that extend Glimmer's mission — and are happy to merge improvements back

## Financial Governance

### Revenue Principles (when applicable)
- No advertising revenue that requires user profiling
- No data sales revenue, ever
- Acceptable: Companion marketplace fees (clearly disclosed)
- Acceptable: Optional premium features (clearly disclosed; no feature degradation for free users)
- Acceptable: Grants, donations, open-source funding

### Financial Transparency
- Annual financial summary published (when the project has revenue)
- No hidden investors who could compromise the project's mission
- Board has authority to review financial decisions that could affect user privacy

## Amendments to This Document

Changes to governance structure require:

1. Proposal published publicly (30-day consultation minimum)
2. Oversight Board review and approval
3. Community notification
4. Change log entry in this document

Changes that reduce accountability or oversight require supermajority (2/3) of the Oversight Board.
